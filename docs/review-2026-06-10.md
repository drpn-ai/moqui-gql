## What's solid (verified, not just claimed)

- **No SQL injection.** All conditions go through parameterized `EntityCondition` with
  whitelisted search keys + operators (`SearchQueryParser`, `ConnectionResolver`); field names
  come from schema declarations, never user input. Values are typed via
  `EntityDefinition.convertFieldString`.
- **Cursors** are opaque base64url tuples (type-tagged sort value + full composite PK), inner
  components base64-encoded (no delimiter injection); tampered cursors fail decode and surface
  as `INVALID_CURSOR`, no internals leaked.
- **Governor:** coerced variables are resolved *before* the cost walk (no low-default /
  high-execution bypass); cost arithmetic saturates (`sat()`, ceiling 100M) against overflow;
  depth/cost/first/batch-key gates run pre-execution.
- **Fragment spreads ARE costed per spread site** — one reviewer claimed a fragment spread
  twice is counted once; verified wrong: `GovernorInstrumentation.Walk.cost()` recurses into
  the fragment definition at every `FragmentSpread` encounter. Disregard if raised.
- **Scope seam** (`ScopeFilters.apply`) is invoked at every entity-find site: root connections,
  nested loaders, single loaders, by-pk and by-identification fetches (proven by ScopeSeamTests).
- **Throttle gate** is thread-safe (synchronized), debits only allowed+applied queries.
- **Test scaffolding is inert:** `echo#GqlField` / `ensure#TestShipments` are not schema-wired
  (`resolver-service` grep clean) and not `allow-remote`.
- **Payment exposure is minimal:** OrderPaymentPreference exposes only id, type, maxAmount,
  statusId — no card data, no encrypted fields, no cost/margin fields anywhere in the schema.
- **View entities** `OrderBillToCustomer` / `FacilityOriginAddress` have correct joins
  (role-typed, date-filtered on both FCM and FCMP); no cartesian-product risk.
- **Transaction timeout** (`gql.queryTimeoutSeconds`) wired via `runUseOrBegin` →
  JDBC statement timeout.
- Every finding from `review-2026-06-03.md` was genuinely addressed in the shipped code.

## For team discussion

### 1. Party PII vs product-store scoping — the one real design question
`ProductStoreScopeFilter` (`scope/ProductStoreScopeFilter.groovy:18`) restricts only entities
that have a `productStoreId` field — documented behavior. But the `parties` root
(`graphql/OmsSchema.gql.xml:161`) is searchable by `firstName`/`lastName` and its backing view
(`co.hotwax.party.party.PartyNameAndRoleDetail`) has no `productStoreId`, so a caller scoped to
one store can enumerate **all parties across stores** in that DB. Same shape for
`FacilityOriginAddress` (lower severity — warehouse addresses, not customer PII).

Fine under the one-DB-per-client precondition with single-store clients. For multi-store
clients, decide one of:
- scope Party via a ProductStore→Party association lookup,
- restrict the `parties`/`party` roots per caller profile, or
- explicitly accept and document the exposure.

### 2. Unbounded preparsed-document cache — FIXED (fix/preparsed-document-cache-cap)
`CachingPreparsedDocumentProvider` keys Moqui cache `gql.preparsed.document` on the **raw query
string**. Confirmed: unconfigured Moqui caches get no size cap (`CacheFacadeImpl.groovy:291` —
"use underlying defaults"). An authenticated caller sending unique query strings grows the cache
without bound (cache-busting memory DoS).
**Fix:** add a `<cache name="gql.preparsed.document" max-elements="..." eviction-strategy="..."/>`
entry to component `MoquiConf.xml`.
The `gql.throttle.bucket` cache has the same property but is keyed by `userId` — bounded by the
real user population; low risk; idle-TTL already acknowledged as future work in `throttle.md` §9.

### 3. GqlQueryLog has no retention story — ADDRESSED (query-log v2, feature/query-log-v2)
Every query (allowed or rejected) writes a row with up to 4000 chars of query text; **no purge
job exists** anywhere in the component. Variables are NOT logged (good), but inline literals in
query text (e.g. `firstName:` searches) do land in the table. Needs a scheduled purge service or
a documented retention decision before real traffic.
Related nit: `writeQueryLog` hardcodes `callerProfile: "default"` (`GqlEngine.groovy:172`) even
when a real profile resolved — the log can't tell which profile governed a query.

### 4. Anonymous SDL endpoint — confirm conscious sign-off
`GET /rest/s1/graphql/sdl` is `anonymous-all` (`service/gql.rest.xml:7`; audit WARN). Commented
as intentional for tooling, but it publishes the full queryable surface (entities, search keys,
sort keys) to unauthenticated callers. Team should explicitly accept.

## Smaller items (fine as documented limitations)

- **Wall-clock budget granularity:** `gql.wallClockBudgetMs` is checked only *between* field
  fetches (`GovernorInstrumentation.beginFieldFetch`); a single slow statement is bounded only
  by the 20s JDBC timeout. Worth a sentence in `design.md`.
- **Nested intra-group pagination tie behavior:** `NestedConnectionLoader.groovy:125` compares
  stringified composite intra-group keys lexicographically — correct only if intra-group keys
  are unique per parent; no test covers non-unique keys. Add a test or document the uniqueness
  assumption. Also the cursor-decode failure there is silently swallowed
  (`catch (Exception ignored)`) — log at WARN for observability.
- **ServiceBackedLoader timeout inheritance:** no current schema users (post-#35); if
  service-backed fields return, verify `ec.service.sync()` honors the transaction deadline or
  pass an explicit timeout.
- **Hygiene (audit INFOs):** missing `<description>`/`short-alias` on the 5 entities in
  `entity/GqlEntities.xml` and `<description>` on 4 services (`GqlExampleServices.xml`,
  `gql.rest.xml`). Five minutes of polish.
