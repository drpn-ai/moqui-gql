# Design Package Review ‚Äî 2026-06-03

Independent multi-lens review of the moqui-gql requirements/design package (4 fresh-context
reviewers: consistency, architecture/feasibility, GraphQL-spec + Shopify conformance,
requirements-coverage/risk). Reviewers verified claims against the live `graphql-java:25.0`
bytecode, the Moqui framework source, and shopify.dev docs.

**Consensus verdict: GREENLIGHT WITH CONDITIONS.** The architecture is sound and buildable; the
two scariest technical bets verified as correct (graphql-java 25.0 instrumentation runs *before*
any DB hit; synchronous DataLoader same-thread dispatch *does* batch in 25.0). The
examples-as-contract discipline and reverse-engineered requirements are strengths. But the cost
governor ‚Äî "the spine" ‚Äî has defects that let pathological queries through, the SDL/examples are
inconsistent about connections vs lists (some example queries are invalid GraphQL), and there are
coverage gaps a real consumer hits immediately. None require a redesign; all are fixable on paper
before build.

Disposition tags: **[FIX]** clear correction (applying) ¬∑ **[DECIDE]** needs a call ¬∑ **[ACCEPT]** noted/accepted risk.

## P0 / P1 ‚Äî Cost governor (the core thesis)

- **[FIX] C1 ‚Äî `maxFirst` vs SDL defaults vs examples contradict.** SDL `picklistItems`/`items`
  default `=500`; examples B4 (`first:200`) and H (`first:500`) shown as success ‚Äî but `maxFirst=100`
  and Task 11 rejects `first>100`. Task 14 would assert success of queries Task 11 rejects.
  ‚Üí reconcile: per-edge `first` caps; lower defaults to ‚â§ the edge cap; fix B4/H.
- **[FIX] C2 ‚Äî Example cost numbers don't compute against the Task 5 formula.** A1 shows
  `requestedQueryCost: 64` but `orderItems(first:50){ fulfillmentStatus(service,25) }` ‚âà 1,600 ‚Üí
  would exceed `maxCost:1000` and be **rejected**, yet A1 is the flagship "success" case.
  ‚Üí make `extensions.cost` values **illustrative, not asserted**; do real cost calibration in
  tests (Task 5/cost-calibration), not hand-set numbers; reconcile whether A1 is allowed (raise
  budget / cost service fields per-level) or trim the flagship query.
- **[FIX] C3 ‚Äî Static cost is `int`; overflows ~2^31.** Three nested `first:1000` = 10^9 (just
  under MAX_INT); a 4th level or larger `first` wraps negative ‚Üí query passes the gate.
  ‚Üí saturating `long` arithmetic clamped below MAX_INT; adversarial test with raw cost >2^31 and
  every `first` ‚â§ cap.
- **[DECIDE/FIX] C4 ‚Äî Service-backed field inside nested lists: cost + blast radius.** Ambiguous
  whether costed per-row (fan-out) or per-level (batched). The service itself issues its own
  queries the analyzer can't see; `queryTimeout` only covers the executor's finds, not the
  service's. ‚Üí cost per-level when batched **plus** a hard batch-key cap (reject when a batched
  service level would receive > K keys); calibrate `serviceFixedCost`; bar service fields in lists
  with no deadline-honoring batch form. Add the missing adversarial example.
- **[FIX] C5 ‚Äî Per-edge `first` cap not enforced on nested edges; no test.** Only root `orders`
  shows `FIRST_REQUIRED`. Cross-product (`orders√óshipGroups√óorderItems` defaults) = 10^6 nodes at
  depth 4 (< maxDepth 6). ‚Üí per-edge max in the schema layer; adversarial tests for nested
  oversized `first`, sibling-fan-out summation (confirm cost sums siblings AND multiplies depth),
  and a declared-but-**unindexed** search key (governance layer 3 has zero test coverage today).
- **[FIX] C6 ‚Äî Wall-clock guard is claimed in phase 1 but deferred in the plan.** design.md lists
  per-request wall-clock as a phase-1 guard; plan defers hard interruption to phase 2 (only
  records elapsed). Many individually-fast batched statements blow total wall-clock under
  per-statement `queryTimeout`. ‚Üí align docs: implement a real per-request deadline in phase 1 **or**
  downgrade the design claim. (Leaning: downgrade claim + add a simple per-request deadline check between levels.)

## P1 ‚Äî GraphQL conformance / SDL

- **[DECIDE] G1 ‚Äî Connection vs plain-list is inconsistent; some example queries are invalid GraphQL.**
  `Order.orderItems`/`shipGroups` are `*Connection!` but A1/B2/L1 select them as plain lists
  (no `edges{node}`); ¬ßN1 selects `edges{node}` on `adjustments` which is a plain list. The
  "shorthand convention #5" is not real GraphQL. Need ONE rule (see decision below), then fix all
  examples + the SDL to match. **Blocking for a valid contract.**
- **[FIX] G2 ‚Äî `sortKey`/`reverse` missing on `pickupOrders`/`returns`/`transferOrders`** (they
  have `query` + `@search` but no sort); **`last`/`before` missing on `Order.returns`**;
  **`ReturnSortKey` defined but never wired**. ‚Üí add sort args (or document exemption); wire or delete `ReturnSortKey`.
- **[FIX] G3 ‚Äî `Decimal` serialized two ways.** Money renders as string (`"129.00"`) per the
  scalar contract, but ¬ßE/¬ßH render `atp`/`quantityOnHandTotal`/`variance` as bare numbers. ‚Üí
  render all `Decimal` as strings (or retype genuinely-integer quantities to `Int`).
- **[FIX] G4 ‚Äî `RELEVANCE` sort key has no meaning in a DB-backed graph** (no full-text). ‚Üí remove
  from enums (or define as a concrete deterministic order).
- **[ACCEPT/FIX] G5 ‚Äî Custom directives (`@search`) are not exposed by standard introspection.**
  The D-A justification ("agents introspect allowed keys") leans on this. ‚Üí ALSO publish the
  search grammar (keys + comparators) in each connection field's **description** (which IS
  introspectable); verify graphql-java serves directives if we rely on them. Add an example of the
  agent introspect‚Üícompose‚Üíreject‚Üíretry loop.

## P1/P2 ‚Äî Architecture / executor

- **[FIX] A1 ‚Äî Keyset cursor predicate.** Plan's `WHERE (sortField,pk) > cursor` (tuple) isn't
  expressible in `EntityCondition` and mishandles NULL/non-unique sort values ‚Üí skipped rows
  (violates ¬ßM "no skip"). ‚Üí OR-form `sort>X OR (sort=X AND pk>Y)`; forbid nullable sort keys (or
  NULLS-LAST emulation); test non-unique + null + backward paging.
- **[FIX] A2 ‚Äî Composite-PK cursors + multi-column `IN` batching under-specified.** `OrderItem` etc.
  are composite-keyed; cursor "pk" is singular; multi-column `IN` isn't standard `EntityCondition`.
  ‚Üí encode full composite PK in cursor; build batch as bounded OR-of-AND per parent, or restrict
  batched edges to single-column FKs in phase 1.
- **[FIX] A3 ‚Äî Batch-cardinality test missing.** 25.0 batches only if every list edge resolves via
  `DataLoader.load` (not a direct find) under `AsyncExecutionStrategy`; one direct-find fetcher
  silently degrades to N+1. ‚Üí keep AsyncExecutionStrategy; add a test asserting exactly-N batch
  loads for N-level nesting.
- **[FIX] A4 ‚Äî `callUseOrBegin` does not exist** ‚Üí `runUseOrBegin`. "Read-only transaction"
  overstates (no JDBC read-only mode) ‚Äî reword or set `Connection.setReadOnly(true)` on the clone.
- **[FIX] A5 ‚Äî Read-replica pool isolation is the "primary control" but ships as an unconfigured,
  untested fallback knob.** ‚Üí make the capped clone pool a checked deliverable (sample
  `<datasource>` with `pool-max`); startup log/assert if GraphQL routes to the base group
  (isolation NOT active); promote a pool-exhaustion smoke test out of "E2E later."
- **[ACCEPT] A6 ‚Äî Read-your-writes staleness on the replica.** Internal app create-then-read may
  read stale. ‚Üí document the eventually-consistent contract; offer `useClone(false)` opt-out for
  trusted read-after-write call sites.
- **[ACCEPT] A7 ‚Äî graphql-java 25.0 instrumentation API verified present** (not deprecated). Keep
  "verify API" as a smoke test; note version coupling to `mantle-shopify-connector`.

## P1/P2 ‚Äî Requirements coverage

- **[DECIDE] R1 ‚Äî No `shipments` list/search root** (Shipment is by-id only). A shipment queue is
  table-stakes for an OMS. Same shape gap: `picklists`, `cycleCounts`, `purchaseOrders` lists.
- **[DECIDE] R2 ‚Äî No party/customer search root** (Party by-id only). The named AI-agent flow
  ("find this customer's orders") needs to resolve a customer first.
- **[DECIDE] R3 ‚Äî Inventory/ATP is point-query only** (single product √ó facility). Bulk
  multi-SKU/multi-facility ATP is one of the most-requested OMS reads.
- **[ACCEPT] R4 ‚Äî Payment folded into `Order.paymentPreferences`** (no standalone) ‚Äî deliberate; note it.

## Strategic

- **[DECIDE/ACCEPT] S1 ‚Äî `query:` string (D-A) is worse for the #1 consumer (AI agents).** The doc
  is honest about this. Mitigations: publish grammar in descriptions (G5); add the retry-loop
  example; consider a typed `filter` for the *agent profile* in phase 2 (the rejected hybrid). ‚Üí
  keep D-A for phase 1, strengthen the agent mitigation, revisit per-profile in phase 2.
- **[FIX] S2 ‚Äî One-DB-per-client is an assumption the layer trusts but can't enforce** (org already
  runs a multi-tenant `unigate`). ‚Üí write it as a checkable precondition ("do NOT enable on a
  shared-DB instance"); add a phase-1 smoke test that the scope-filter seam is invoked.
